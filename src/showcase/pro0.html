旅游 Agent 项目

对话式旅游规划助手：先理清需求，再给出可执行的行程建议。

一、产品

旅游 Agent 面向「自由行 / 跟团」等常见出行场景，帮助用户把零散想法整理成可阅读、可调整的行程方案：从目的地、天数、预算与人数出发，逐步补齐交通、住宿、玩法偏好等关键信息，再输出分日安排与注意事项，减少反复沟通与手工整理成本。

与单次问答不同，产品侧强调可持续对话：同一用户可在多轮对话里补充约束、替换偏好、迭代方案，让结果随真实需求演进，而不是「一问一答就结束」。

三、技术方案

技术栈（落地选型）：后端以 Python 为主；大模型侧统一走 LangChain（ChatOpenAI 等绑定、LCEL Runnable、Tools / Tool calling、多轮消息拼装）；Web 与 CLI 共用同一套服务，HTTP 层用 FastAPI，对话流式输出走 SSE。记忆召回用 Chroma 向量库存摘要与偏好片段；会话流水、消息与 DAG 任务状态落在 SQLite。联网事实核查通过 LangChain 封装的 检索 / Search Tool 调用（具体 Provider 与 API Key 由配置决定）。

模型：与 OpenAI 协议兼容的 Chat 接口；开发与控成本时常用 gpt-4o-mini，行程结构复杂或长上下文场景可切到 gpt-4o（或同端点映射的国产模型），模型名与 base_url、密钥均走环境变量 / .env，便于本地与部署环境切换。

子 Agent 分工（名称与职责）：

  • 槽位 Agent：从自然语言里抽取目的地、日期区间、预算、人数、交通/住宿偏好等；缺槽则生成追问，补全后再放行。
  • 规划 Agent：把已确认槽位拆成可并行子任务，写出依赖关系（DAG）并落库，供下游执行与断点恢复。
  • 景点与玩法 Agent：专注日程与景点逻辑，按需挂载搜索 Tool，补齐开放时间、闭馆日等可变信息。
  • 交通 / 住宿 Agent：分别或合并为「资源」子 Agent，负责票、酒、接驳等结构化建议（同样可走 Tool，结果回写任务节点）。
  • 汇总 Agent：收集各子任务输出，去重冲突、统一口径，生成最终 Markdown 行程稿（分日 + 注意事项）。
